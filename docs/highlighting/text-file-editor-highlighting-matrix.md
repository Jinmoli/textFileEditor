# Text File Editor 高亮能力矩阵

基于当前仓库实现，`xml / yml / yaml / sql` 主要依赖 CodeMirror 官方语言包，`ps1` 依赖 legacy PowerShell mode，`sh / bash / zsh / bat / cmd` 依赖仓库内自定义 `StreamParser`。因此当前高亮治理的重点不是替换全部语言来源，而是优先补强脚本类 parser 的规则和回归样例。

| 类型 | 当前语言来源 | 当前优点 | 主要问题 | 建议动作 | 优先级 |
| --- | --- | --- | --- | --- | --- |
| `xml` | `@codemirror/lang-xml` | 标签、属性、字符串等基础结构高亮稳定，维护成本低 | 复杂命名空间、长属性串、混合内容场景还缺少仓库内样例验证 | 保持现有实现，补充真实 XML 样例做人工验收和回归检查 | `P1` |
| `yml / yaml` | `@codemirror/lang-yaml` | 键值、列表、注释等常见 YAML 结构已具备成熟支持 | 多行文本、锚点、别名、复杂缩进场景是否足够清晰还未验证 | 保持现有实现，增加覆盖锚点、别名、多行块文本的样例矩阵 | `P1` |
| `sql` | `@codemirror/lang-sql` | 主流 SQL 关键字和常见结构已有现成高亮能力 | 不同数据库方言、函数名、复杂 DDL/DML 的可读性还不明确 | 先建立真实 SQL 样例库，必要时再评估补充关键字映射或方言配置 | `P1` |
| `sh / bash / zsh` | 仓库内自定义 `StreamParser` | 已覆盖常见关键字、变量、数字、字符串、基础操作符，落地简单 | 命令替换、数组、here-doc、函数声明细节等高级 shell 语法漏高亮风险较高 | 优先补充脚本样例并扩展 token 规则；如后续复杂度继续上升，再评估迁移到更完整的语言支持 | `P0` |
| `bat / cmd` | 仓库内自定义 `StreamParser` | 已覆盖 `rem/::` 注释、变量、部分关键字和基础操作符 | 标签跳转、延迟变量、复杂条件分支、批处理特殊语法覆盖不完整 | 优先补强 parser 规则并建立批处理回归样例，减少明显漏高亮 | `P0` |
| `ps1` | `@codemirror/legacy-modes` 的 `powershell` mode | 基础 PowerShell 语法已有现成支持，优于从零自定义 | 高级函数、特殊运算符、复杂脚本块的实际显示效果仍需样例验证 | 先补 PowerShell 样例验证，不急于重写语言支持 | `P1` |

## 核心结论

1. `xml / yml / yaml / sql / ps1` 当前主要问题不是“没有语言支持”，而是“缺少真实样例验证”。
2. `sh / bash / zsh / bat / cmd` 当前风险最高，因为它们依赖仓库内轻量自定义 parser，能力边界直接受现有关键字和正则规则限制。
3. 后续治理建议先做“样例矩阵 + 回归检查”，再决定是否需要引入更重的脚本语言支持方案。
